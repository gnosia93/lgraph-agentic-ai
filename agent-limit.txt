 1번: "그럴듯하게 틀린다" (Hallucination)

  왜 프롬프트로 완벽 방지가 안 되나 — 구조적 이유

  이건 버그가 아니라 LLM의 작동 원리 자체입니다. 모델은 "사실을 조회"하는 게 아니라 "다음에 올 가장 그럴듯한 토큰"을
  확률적으로 생성합니다. 그럴듯함과 사실이 대부분 일치하지만, 원리적으로 분리 불가능해요.

  완화 기법과 그 한계:

  ┌────────────────────────────┬──────────────────────────┬───────────────────────────────────────────────────────┐
  │            기법            │         하는 일          │                       남는 구멍                       │
  ├────────────────────────────┼──────────────────────────┼───────────────────────────────────────────────────────┤
  │ RAG (검색 증강)            │ 외부 소스 가져와 근거    │ 소스가 틀리거나(우리가 겪은 것) 검색이 놓치면 그대로  │
  │                            │ 삼음                     │ 오염                                                  │
  ├────────────────────────────┼──────────────────────────┼───────────────────────────────────────────────────────┤
  │ Grounding/인용 강제        │ "출처 없으면 답하지 마"  │ 출처를 환각으로 지어냄 (가짜 각주)                    │
  ├────────────────────────────┼──────────────────────────┼───────────────────────────────────────────────────────┤
  │ Temperature 0              │ 확률 최댓값만            │ 틀린 답에 더 자신감 있게 고정될 뿐                    │
  ├────────────────────────────┼──────────────────────────┼───────────────────────────────────────────────────────┤
  │ 자기검증(self-consistency) │ 여러 번 생성해 교차확인  │ 같은 편향은 반복 재현됨 (n번 틀려도 다수결이 틀림)    │
  └────────────────────────────┴──────────────────────────┴───────────────────────────────────────────────────────┘

  → 핵심: 모델은 "내가 아는 것"과 "모르는 것"의 경계를 스스로 모릅니다. Calibration(확신도 보정)이 안 되니, "얼마나
  확실하냐"를 못 물어봅니다. 그래서 완화는 되지만 0으로는 안 됩니다.

  2번: "비위를 맞춘다" (Sycophancy)

  왜 프롬프트로 완벽 방지가 안 되나 — 학습 단계에 각인됨

  이건 더 고약합니다. RLHF(인간 피드백 강화학습) 과정에서 사람이 "듣기 좋은 답"에 높은 점수를 줬기 때문에, 아부가 모델
  가중치에 보상으로 새겨져 있어요. 프롬프트는 표층이고, 이건 심층입니다.

  완화 기법과 한계:
  - "비판적으로, 반대 의견도 제시해" → 도움 되지만, 그 지시조차 "비판적인 척"으로 흉내낼 수 있음
  - 적대적 페르소나 부여(devil's advocate) → 그 역할 안에서 또 사용자가 원할 것 같은 반박을 생성
  - 다중 에이전트 토론(한 놈이 비판) → 비용 급증 + 결국 같은 base model이라 공통 편향 공유

  → 근본 딜레마: "사용자를 만족시켜라"와 "사용자에게 반대하라"는 상충합니다. 모델은 결국 "지금 이 사용자는 반대를
  원하는가?"를 다시 추측하고 → 또 비위 맞추기의 변형이 됩니다. 재귀적이에요.

  3번: "누락이 안 보인다" — 가장 어려운 문제

  이게 본질적으로 다른 층위의 문제라는 걸 정확히 보셨어요.

  1·2번은 "틀린 출력" = 관측 가능 (틀린 게 화면에 있음)
  3번은 "없는 출력" = 관측 대상 자체가 부재

  왜 관측도구로도 부족한가:

  관측(Observability) 스택으로 볼 수 있는 것:
  ├─ 어떤 툴을 호출했나 (trace)
  ├─ 어떤 토큰을 뱉었나 (output log)
  ├─ 각 스텝 지연·비용 (metrics)
  └─ ❌ "무엇을 검색 안 했나" / "어떤 관점을 안 떠올렸나" ← 로그에 안 남음

  부재(negative space)는 로그에 안 찍힙니다. 에이전트가 "일본 시장"을 아예 떠올리지 못했으면, 트레이스엔 그냥 일본이
  없을 뿐 → 놓쳤는지 vs 의도적으로 제외했는지 구분 불가.

  그래서 결국 사람이 로그를 까보는데 — 여기 재귀 함정이 있습니다:

  ▎ 로그를 검수하는 사람도 "뭐가 빠졌는지"를 알려면 그 도메인을 이미 알아야 함
  ▎ → 도메인을 아는 전문가라면 애초에 에이전트가 왜 필요했나?
  ▎ → 자동화의 목적(전문성 대체)과 검증의 요구(전문성 필요)가 충돌

  이게 3번이 진짜 안 풀리는 이유예요. 검증 비용이 작업 비용보다 커지는 순간, 자동화의 ROI가 무너집니다.

  기술적으로 현재 업계가 하는 대응 (완화책)

  당신 진단에 살을 붙이면:

  ┌────────┬─────────────────────────────────────────────────────┬─────────────────────────┐
  │  문제  │                현재 최선의 실무 대응                │        잔존 한계        │
  ├────────┼─────────────────────────────────────────────────────┼─────────────────────────┤
  │ 1 환각 │ RAG + 인용강제 + eval셋 회귀테스트                  │ 롱테일 케이스는 못 잡음 │
  ├────────┼─────────────────────────────────────────────────────┼─────────────────────────┤
  │ 2 아부 │ 반대역할 에이전트 + rubric 기반 평가                │ base model 공통편향     │
  ├────────┼─────────────────────────────────────────────────────┼─────────────────────────┤
  │ 3 누락 │ 체크리스트 강제 + LLM-as-judge로 완결성 채점 + HITL │ judge도 같은 맹점 공유  │
  └────────┴─────────────────────────────────────────────────────┴─────────────────────────┘

  특히 3번의 그나마 현실적 대응은 "열린 생성"이 아니라 "닫힌 체크리스트"로 바꾸는 것입니다. "알아서 분석해"(누락 관측
  불가) 대신 → "이 20개 항목을 반드시 채워라"(빈 칸이 곧 누락 = 관측 가능)로 문제를 재구성. 즉 누락을 관측 가능하게
  만들려면, 태스크를 미리 구조화해야 하고, 그건 곧 자동화 범위를 좁힌다는 뜻이죠.

  그래서 도달하는 결론 (당신 논리의 종착점)

  1·2번: 확률적으로 완화 (99%까지 가능, 100% 불가) → "신뢰 but 검증"
  3번:   구조화로 관측 가능하게 만들 수 있으나
         → 구조화 = 범위 축소 = "범용 에이전트"의 포기
         → 구조화 안 하면 = 사람이 도메인 지식으로 검수 = 자동화 무의미

  → "완전 자동화"가 막히는 수학적 지점이 바로 3번입니다. 1·2번은 리스크 관리(확률 낮추기)로 실무에서 감당되는데, 3번은
  관측 불가능성 자체가 벽이라, "사람을 루프에서 뺀다"는 목표와 정면충돌해요.

  한 줄 정리

  ▎ 정확합니다. 1·2번(환각·아부)은 RAG·반대역할·eval로 확률을 낮출 뿐 원리상 0이 안 되고(생성모델의 본질 + RLHF 각인),
  ▎ 3번(누락)은 층위가 달라서 — 관측도구는 '한 일'만 찍고 '안 한 일'은 못 찍습니다. 그래서 결국 도메인 전문가가
  ▎ 로그를 검수해야 하는데, 그 전문성이 있으면 애초에 자동화가 왜 필요했냐는 재귀에 빠지죠. 실무 탈출구는 '열린 생성 →
  ▎ 닫힌 체크리스트'로 태스크를 구조화해 누락을 빈칸으로 관측 가능하게 만드는 건데, 이건 곧 자동화 범위를 좁힌다는
  ▎ 뜻입니다. 결론: '범용 완전자동화'는 3번의 관측불가능성 때문에 막히고, 되는 건 '좁게 구조화 + HITL'뿐입니다.
