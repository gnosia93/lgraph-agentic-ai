## RULE 기반의 평가 ##

현재 LLM 평가 아키텍처에서 Rule 기반 평가(코드/결정론적 알고리즘 기반)는 크게 두 가지 갈래로 나뉜다.

### 1. N-gram 기반의 통계적 평가 (어휘 유사도) ###
N-gram(연속된 ‭N‬개의 단어/토큰 뭉치) 방식을 쓰는 지표들로, 모델의 응답과 정답(Ground Truth)의 텍스트가 얼마나 표면적으로 일치하는지 통계적으로 계산한다

* BLEU (Bilingual Evaluation Understudy):
기계 번역 평가에 주로 쓰이며, 모델 응답의 N-gram이 정답에 얼마나 포함되어 있는지 정밀도(Precision) 중심으로 계산.

*	ROUGE (Recall-Oriented Understudy for Gisting Evaluation):
요약 작업 평가에 주로 쓰이며, 정답의 N-gram을 모델이 얼마나 놓치지 않고 가져왔는지 재현율(Recall) 중심으로 평가한다. (※ 단, ROUGE-L은 N-gram이 아니라 가장 긴 공통 부분 문자열(LCS) 알고리즘을 사용)
비용이 0원에 가깝고 매우 빠르며 결정론적(Deterministic)이지만, 의미는 전혀 모른 채 글자 매칭만 하기 때문에 패러프레이징(말 바꾸기)이나 문장 구조가 바뀌면 점수가 떡락하는 한계가 있다.

### 2. N-gram을 쓰지 않는 프로그래밍 방식의 Rule 기반 평가 ###
최근 프롬프트푸(Promptfoo)나 에비던트리(Evidently AI) 같은 LLM 평가 툴을 세팅할 때 코드 레벨에서 거르는 Reference-free(정답 예시가 필요 없는) 검증 알고리즘들도 모두 Rule 기반 평가에 속한다. 
이들은 N-gram을 쓰지 않는다.

* 정규표현식(Regex) 검사: 응답에 이메일, 전화번호, 주민번호 같은 개인정보(PII) 패턴이 포함되었는지 혹은 필수 키워드가 들어갔는지 검사.
* 스키마/포맷 벨리데이션: 모델에게 JSON, SQL, XML 출력을 요구했을 때, json.loads()를 통과하는지(구문 오류가 없는지), 지정한 Key(구조)가 누락 없이 파싱되는지 코드로 검사.
* 코드 실행 및 통과 여부 (Code Execution): LLM이 뱉은 파이썬 코드를 샌드박스 환경에서 실제로 실행하여 Unit Test를 통과하는지 검사.
* 길이 및 메타데이터 제한: 응답 글자 수가 너무 길지 않은지, 금지어가 포함되어 있지 않은지 검사.

#### 🛠️ 요약하자면 ####
*	`정답 문장과 얼마나 똑같이 썼는가?` 를 보는 Rule 기반 평가는 N-gram 기준(BLEU, ROUGE 등)이 맞다.
* `출력 형식이나 보안 규칙을 잘 지켰는가?`를 보는 Rule 기반 평가는 정규식, 스키마 검증, 코드 실행 등 프로그래밍적 룰을 사용한다.
  
현업에서는 이 두 가지 Rule 기반 평가로 1차 스크리닝(Smoke Test)을 저비용으로 끝낸 뒤, 문맥과 뉘앙스를 보는 임베딩 기반 평가(BERTScore)나 LLM-as-a-judge (G-Eval 등)를 2차로 얹어서 앙상블로 평가 아키텍처를 가져간다.

## BERTScore ##

BERTScore는 LLM이 뱉은 응답과 정답(Ground Truth) 문장을 비교하는 것으로, N-gram(BLEU)처럼 글자 매칭을 하는 게 아니라, BERT 같은 임베딩 모델을 사용해 두 문장을 벡터로 바꾼 후 '문맥적 의미의 유사도(Cosine Similarity)' 를 계산한다.
예를들어 정답이 "나는 배가 고프다"이고 LLM 응답이 "나 출출해"라면, 글자는 하나도 안 겹치지만 BERTScore는 두 단어의 임베딩 벡터가 유사하므로 높은 점수를 보여준다.
주로 번역, 요약, 챗봇 응답이 정답과 얼마나 의미가 일치하는지 평가할 때 사용된다. (정답 예시 데이터셋이 필수)

## Perplexity (PPL) ##

Perplexity는 문장의 '유창함(Fluency)'과 '예측 가능성'을 측정하는 지표입니다. 수학적으로는 "다음에 올 단어를 예측할 때 모델이 얼마나 헷갈려(당황해) 하는가"를 나타낸다.
정답 문장이 따로 필요 없고, 모델이 문장을 생성할 때, 각 단어를 뱉을 확률(Probability)의 역수를 기하평균 낸 값이다.
점수가 낮을수록 좋다.

* PPL이 10이다 = "이 모델은 다음에 올 단어 후보를 10개 정도로 좁혀서 명확하게 확신하고 있구나 (똑똑하네)."
* PPL이 100이다 = "다음에 무슨 단어를 써야 할지 몰라 100개 사이에서 엄청 헤매고 있구나 (어색한 문장이네)."

LLM의 사전 학습(Pre-training) 단계나 파인튜닝 단계에서 모델 자체의 기본 언어 능력이 얼마나 고도화되었는지 평가할 때 사용한다.
우리가 일상적으로 만드는 'LLM 애플리케이션(RAG, 에이전트 등)의 서비스 평가 파이프라인'에는 PPL(Perplexity)을 넣지 않는 것이 정석이다. 

### 1. 현업 서비스 평가 파이프라인에 PPL을 안 넣는 이유 ###
promptfoo, Ragas, DeepEval 같은 서비스 평가 인프라를 구축할 때 PPL을 제외하는 이유는 "우리가 통제할 수 없는 지표"이기 때문이다.

*	API 모델(OpenAI, Anthropic 등)은 PPL 계산이 불가능: PPL을 계산하려면 모델이 단어를 뱉을 때 내부적으로 계산한 '확률값(Logits)'을 알아야 하는데 GPT-4o나 Claude 같은 클로즈드 소스 모델들은 API로 완성된 텍스트 결과만 툭 던져줄 뿐, 내부 확률 데이터(Logprobs)를 주지 않는다.

*	비즈니스 목적에 맞지 않습니다: PPL은 "이 문장이 문법적으로 유창한가?"만 본다. LLM이 아주 유창하고 완벽한 문법으로 거짓말(환각)을 하거나, "네, 저는 바보입니다"라고 헛소리를 해도 문장이 매끄러우면 PPL 점수는 아주 좋게(낮게) 나옵니다. 우리는 '비즈니스 지시를 잘 따랐는지(Instruction Following)', '정확한 정보를 줬는지(Faithfulness)'가 궁금한데, PPL은 이를 잡아내지 못한다.

### 2. 그럼 PPL은 대체 어떤 파이프라인에 들어가는 걸까? ###

PPL이 주인공으로 활약하는 파이프라인은 따로 있는데, 바로 [LLM 모델 자체를 만드는(만들고 튜닝하는) 파이프라인] 에 사용된다.

* LLM 사전 학습(Pre-training) / 파인튜닝(Fine-tuning) 파이프라인: Llama 3나 오픈소스 모델을 가져와 우리 회사 데이터로 추가 학습(파인튜닝)을 시킬 때, 모델이 헛소리를 하지 않고 언어 능력을 제대로 습득하고 있는지 배치(Batch)마다 Loss 값과 함께 PPL 파이프라인을 돌려 모니터링한다. (가중치와 Logits에 직접 접근이 가능)

* 인프라/서빙 최적화 파이프라인: 모델을 양자화(Quantization, FP16 ➡️ INT8 등으로 경량화)할 때, 모델 덩치를 줄였는데도 언어 이해 능력이 망가지지 않았는지(PPL이 튀지 않는지) 검증하는 파이프라인에 필수적으로 들어간다.

## LLM 평가 파이프라인 ##
LM 평가는 단순한 '시험 채점'이 아니라, [데이터 주입 ➡️ 모델 실행 ➡️ 다단계 검증 ➡️ 시각화]까지 일련의 자동화된 공정 라인을 설계하는 작업이기 때문에 이를 평가 파이프라인(Evaluation Pipeline)이라고 부른다. 
```
[테스트 데이터 세팅] 
      │
      ▼
[LLM API 호출 및 응답 세대(Generation)]
      │
      ▼
[0차 스모크 테스트] (JSON 파싱 성공 여부, Regex 패턴 매칭)
      │
      ▼
[1차 통계적 평가] (정답 예시와 N-gram/BLEU 유사도 비교)
      │
      ▼
[2차 의미론적 평가] (임베딩 모델을 통한 벡터 유사도/BERTScore 계산)
      │
      ▼
[3차 LLM-as-a-judge] (GPT-4 등을 이용한 환각/지시 이행 점수 산출)
      │
      ▼
[결과 취합 및 대시보드 시각화] (Grafana, Promptfoo UI 등)
```
