워크샵 제목: 프로덕션 환경을 위한 LLM & 에이전틱 AI 평가 인프라 구축 실무
부제: 감(Intuition)에 의존하는 테스트에서 숫자로 증명하는 엔지니어링 파이프라인으로

1부: 왜 지금 LLM 평가 인프라인가? (The Architecture)

•	1.1 에이전틱 AI 시대의 새로운 리스크: 챗봇의 거짓말과 에이전트의 오작동(액션 실패)이 가져오는 치명적인 차이
•	1.2 비결정론적 카오스 제어하기: 프롬프트 튜닝과 모델 버전 업 업그레이드 시 발생하는 성능 역전(Regression) 현상
•	1.3 비용(Cost)과 속도(Latency)의 트레이드오프: 프론티어 모델(GPT-4/Claude)과 가성비 경량 모델(SLM)의 하이브리드 아키텍처 설계 전략

2부: 평가 파이프라인의 핵심 컴포넌트 분해 (The Metrics)

•	2.1 [0차 필터] 스모크 테스트(Smoke Test) 구축:
•	비싸고 느린 LLM 채점 전에 정규식(Regex)과 스키마 검증으로 불량 응답 빠르게 걷어내기
•	2.2 [1차 필터] 결정론적 규칙(Rule) 및 통계적 평가:
•	어휘 유사도 측정의 한계와 이해: N-gram(BLEU, ROUGE)이 여전히 유용한 순간
•	2.3 [2차 필터] 의미론적(Semantic) 유사도 평가:
•	글자가 틀려도 문맥을 읽는 벡터 임베딩 기반의 평가 (BERTScore 활용법)
•	Perplexity(PPL) 지표는 왜 서비스 평가 파이프라인에서 제외해야 하는가?
•	2.4 [3차 필터] LLM-as-a-judge 아키텍처:
•	LLM을 판사로 고용해 환각(Hallucination)과 지시 이행 여부를 정량적으로 채점하는 방법

3부: [Hands-on] 우리 서비스 전용 '시험문제 은행'과 자동화 파이프라인 만들기

•	3.1 골든 데이터셋(Golden Dataset) 설계: 오픈소스 벤치마크(MMLU)의 함정 탈출, 유저 로그 기반의 엣지 케이스 30선 구축 기법
•	3.2 오픈소스 평가 프레임워크 연동: Promptfoo / DeepEval을 활용한 멀티 모델(OpenAI vs Anthropic vs Ollama) 동시 벤치마킹 환경 세팅
•	3.3 CI/CD 자동화 파이프라인 통합: 깃허브 액션(GitHub Actions)과 연동하여 프롬프트 변경 시 자동으로 스모크 테스트부터 시각화 리포트까지 쏴주는 아키텍처 완성

💡 워크샵 운영 팁 (Architect's Note)
•	타깃 청중: LLM 서비스를 프로덕션에 배포해야 하는데 "모델 버전 올라가면 기존 프롬프트가 깨지지 않을까?" 불안해하는 시니어 개발자 및 테크 리드.
•	워크샵의 엣지: 단순히 "이 툴 쓰면 좋습니다"가 아니라, "비싼 LLM 평가 전에 정규식과 스모크 테스트로 돈을 아끼고, 임베딩과 LLM 판사를 엮어서 다단계 방화벽 파이프라인을 쳐야 한다"는 엔지니어링적 설계 사상(Architecture)을 강조.
