RAG(Retrieval-Augmented Generation) 시스템의 평가는 일반적인 LLM 평가와 완전히 다릅니다. 단순히 "답변이 그럴싸한가?"를 보는 게 아니라, **1) 문서를 찾아오는 '검색(Retrieval)'**과 **2) 찾아온 문서로 답변하는 '생성(Generation)'**의 두 가지 단계를 철저하게 분리해서 평가해야 합니다.
실무에서 가장 글로벌 표준으로 쓰이는 가이드라인은 RAG Triad(삼형제) 프레임워크입니다. 이를 기반으로 한 자동화 평가 도구(Ragas, DeepEval 등)를 활용해 점수화(‭$0 \sim 1$‬‭‬)하는 것이 정석입니다.

### 📐 1. RAG 평가의 핵심 기준: RAG Triad ###
RAG 파이프라인의 품질은 다음 세 가지 축으로 평가합니다.
```
       [ Query ]
        /     \
       /       \
  Context     Groundedness
 Relevance       /
     /          /
    v          v
[ Context ] ------> [ Response ]
           Answer
          Relevance
```

#### ① 맥락 관련성 (Context Relevance) ####
* 정의: 유저가 던진 질문에 대해, 검색기(Retriever)가 진짜 쓸모 있는 문서들을 잘 찾아왔는가? (노이즈가 얼마나 섞였는가?)
* 평가 방법: 검색된 청크(Context)들 중 질문(Query)에 답변하는 데 직접적으로 기여한 문장의 비율을 LLM이 평가합니다.
* 해결책: 이 점수가 낮다면 임베딩 모델을 바꾸거나, Chunking 전략 수정, 또는 Parent Document Retriever/Reranker(코히어 등)를 도입해야 합니다.

#### ② 근거성 / 환각 여부 (Groundedness / Faithfulness) ####
* 정의: 생성된 답변이 오직 찾아온 문서(Context)의 내용에만 기반하고 있는가? (지어낸 말이나 LLM 자체 지식으로 아는 척하지 않았는가?)
* 평가 방법: 생성된 답변의 문장들을 쪼갠 뒤, 각 문장이 검색된 문서 안에서 추론/증명 가능한지 LLM이 검증합니다.
* 해결책: 이 점수가 낮다면 시스템 프롬프트에 *"반드시 제공된 문서에 기반하여 답변하고, 모르면 모른다고 하라"*는 가드레일을 강하게 쳐야 합니다.

#### ③ 답변 관련성 (Answer Relevance) ####
* 정의: 생성된 답변이 유저의 질문 취지에 맞는 올바른 대답인가? (동문서답하지 않았는가?)
* 평가 방법: 완성된 답변을 보고 역으로 "이 답변이 나올 법한 질문"을 생성한 뒤, 원래 유저의 질문과 유사도를 비교합니다.
* 해결책: 이 점수가 낮다면 LLM 자체의 성능(파라미터 크기)을 올리거나 프롬프트의 Instruction을 명확히 해야 합니다.

### 🛠️ 2. 실무에서 쓰는 대표적인 오픈소스 평가 툴킷 ###
이 3가지 지표를 매번 사람이 눈으로 검사할 수 없기 때문에, '평가용 LLM(주로 GPT-4o 등)'을 감독관으로 세워 자동화합니다.

#### ① Ragas (Retrieval Augmented Generation Assessment) ####
현재 RAG 평가에서 가장 유명한 파이썬 라이브러리입니다.
* 특징: 질문, 검색된 문서, 답변 데이터셋만 Dataframe 형태로 넣어주면 context_precision, faithfulness, answer_relevance 등의 점수를 알아서 매겨줍니다.

#### ② DeepEval / Promptfoo ####
* 특징: CI/CD 파이프라인(GitHub Actions 등)에 태우기 아주 좋게 설계되어 있습니다. 코드가 바뀔 때마다 RAG 점수가 떨어지지 않는지 회귀 테스트(Regression Test)용으로 적합합니다.

### 3. 평가 프로세스 구축 3단계 (실무 가이드) ###
#### 1. Golden Dataset(정답 셋) 구축 (가장 중요) ####
현업 담당자들과 머리를 맞대고 [유저 질문 - 예상되는 정답(Ground Truth)] 세트를 최소 50~100개 이상 만듭니다. (Ragas의 합성 데이터 생성 기능을 이용해 LLM이 문서를 기반으로 질문-답변 쌍을 자동 생성하게 할 수도 있습니다.)

#### 2. 파이프라인 로깅 ####
유저가 질문했을 때 **'어떤 문서 청크들이 검색되었는지(Context)'**와 **'최종 답변(Response)'**을 DB나 랭스미스(LangSmith), 피닉스(Phoenix) 같은 관측성(Observability) 툴에 반드시 로깅해야 합니다.

#### 3. 지표 정량화 및 튜닝 ####
구축된 정답 셋을 툴킷에 돌려 평균 점수(‭$0.85$‬ 이상 등)를 타겟팅합니다.
* 예: "검색 점수(Context Recall)는 높은데 환각 점수(Faithfulness)가 낮네? 프롬프트를 조져야겠다." / "반대로 답변 품질은 좋은데 엉뚱한 문서를 들고 오네? 임베딩이랑 하이브리드 검색(BM25 + Vector)을 섞어야겠다."

RAG 시스템은 무작정 개발하는 것보다 이 평가 판을 먼저 짜놓고 파라미터(Chunk size, Overlap, Top-K, LLM temperature 등)를 미세 조정(A/B Test)하는 것이 전체 개발 기간을 대폭 줄이는 지름길입니다.

RAG 파이프라인을 구축할 때 가장 많이 하는 실수가 **"모든 데이터에 똑같은 텍스트 분할기(RecursiveCharacterTextSplitter)와 기본 임베딩"**을 적용하는 것입니다.
데이터의 형태(Text, Table, Code, Log 등)에 따라 검색기(Retriever)가 멍청해지는 이유가 완전히 다르기 때문에, 데이터 맞춤형 턱밑 처방이 들어가야 합니다. 실무에서 바로 쓸 수 있는 데이터 유형별 검색 품질 최적화 팁을 정리해 드립니다.
1. 정형 데이터 / 표(Table)가 많은 데이터 (금융서, 보고서 등)
•	문제점: 표를 일반 텍스트로 긁으면 문맥이 다 깨집니다. 예를 들어 "2025년 영업이익" 열과 "300억" 행의 연관 관계를 벡터 임베딩이 인지하지 못하고 엉뚱한 숫자를 들고 옵니다.
•	🔥 최적화 팁:
•	Markdown/HTML 변환 파싱: 문서를 읽어올 때 단순 텍스트가 아닌 Unstructured나 LlamaParse 같은 툴을 써서 표를 | 년도 | 영업이익 | 형태의 Markdown 문법이나 HTML Table 구조로 보존해야 합니다. LLM과 최신 임베딩 모델은 이 마크다운 구조를 기가 막히게 이해합니다.
•	Multi-Vector Retriever (요약본 임베딩): 표 자체를 벡터로 만들면 검색 효율이 떨어집니다. LLM에게 표를 먼저 보여주고 *"이 표는 ~년에 대한 ~기업의 재무제표 요약본이다"*라고 텍스트 요약본을 만들게 한 뒤, 임베딩은 요약본으로 하고, 유저가 검색 성공 시 원본 표(Table) 청크를 LLM에 전달하는 구조를 짜야 합니다.
2. 계층 구조 / 법률, 약관, 매뉴얼 (조, 항, 목 구분이 엄격한 데이터)
•	문제점: "제3조 2항에 의거하여~" 같은 문서는 앞뒤 맥락(상위 대주제)을 모르면 하위 청크만 읽었을 때 동문서답하기 십상입니다. 글자 수 기준으로 툭툭 자르면 법적 조항이 반토막 나서 환각의 주범이 됩니다.
•	🔥 최적화 팁:
•	Parent Document Retriever (부모-자식 청크): 검색용 청크는 아주 작게 쪼개서(자식 청크: 200300자) 검색 정확도를 올리고, 막상 LLM에게 컨텍스트를 넘겨줄 때는 그 자식이 속한 **문서 전체나 상위 챕터(부모 청크: 10002000자)를 통째로 뱉어내게** 매핑 딕셔너리를 구현합니다.
•	Metadata Enrichment: 각 청크의 메타데이터에 {"장": "제1장 총칙", "조": "제3조"} 같은 계층 구조 정보를 강제로 주입하고, 유저가 "3조에 대해 알려줘" 하면 벡터 검색 전에 **메타데이터 필터링(Metadata Filtering)**을 먼저 걸어 검색 범위를 압축해야 합니다.
3. 개발 코드(Source Code) 및 API 도큐먼트
•	문제점: 코드는 함수명, 클래스명, 특수문자({, }, =>)가 난무하여 일반 문장용 임베딩 모델(예: OpenAI text-embedding-3)이 유사도를 잘 측정하지 못합니다.
•	🔥 최적화 팁:
•	Language-Specific Splitter: LangChain 등에서 제공하는 Language.PYTHON, Language.JS용 구문 분할기를 사용해야 합니다. 글자 수가 아니라 함수 단위, 클래스 단위로 코드가 깨지지 않게 예쁘게 잘라줍니다.
•	코드 전용 임베딩 및 하이브리드 검색: 코드 검색은 의미론적(Semantic) 검색보다 정확한 함수명 키워드 매칭이 더 중요할 때가 많습니다. 따라서 벡터 검색에만 의존하지 말고, 키워드 기반의 BM25 검색을 5:5 비율로 섞는 하이브리드 검색(Hybrid Search) 후 Reranker를 무조건 태워야 품질이 올라갑니다.
4. 시간에 따라 변하는 데이터 / 로그, 히스토리 데이터
•	문제점: "가장 최신 보안 정책이 뭐야?"라고 물었을 때, 2024년 청크와 2026년 청크의 벡터 유사도가 비슷하다면 옛날 문서를 들고 와서 과거의 답변을 내놓는 대참사가 발생합니다.
•	🔥 최적화 팁:
•	Time-Weighted Vector Store: 메타데이터에 문서 생성/수정 날짜(timestamp)를 무조건 박아두고, 검색 점수를 계산할 때 **최신 날짜일수록 가산점(Recency 보정)**을 주는 스코어링 알고리즘을 적용합니다.
•	Auto-Query Generation (Self-Querying): 사용자가 "최근 정책 알려줘"라고 하면, LLM이 가드레일 단에서 이 질문을 분석해 filter={'date': {'$gte': '2026-01-01'}} 같은 시간 필터 쿼리를 자동으로 생성하여 벡터 DB를 찌르게 아키텍처를 세팅합니다.
💡 실무자를 위한 한 줄 요약 고수칙
데이터 종류를 막론하고 검색 품질을 가장 확실하고 가성비 있게 올리는 치트키는 "하이브리드 검색(Sparse+Dense) ➡️ 앙상블 ➡️ 리랭커(Reranker, 예: Cohere나 BGE-Reranker) 배치" 파이프라인을 구축하는 것입니다. 검색기가 똥같이 찾아와도 리랭커가 상위 3~5개로 기가 막히게 재정렬해 주기 때문에 RAG 삼형제(Triad) 점수가 비약적으로 상승합니다.
