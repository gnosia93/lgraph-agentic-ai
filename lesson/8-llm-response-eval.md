LLM 답변 품질 평가는 크게 자동 평가와 사람 평가로 나뉘고, 실무에서는 둘을 섞어서 씁니다. 대표적인 방식들을 정리해드릴게요.

기준(무엇을 평가하나) 먼저 "품질"을 여러 축으로 나눕니다.

* 정확성(Accuracy / Factuality): 사실이 맞는지, 환각(hallucination)이 없는지
* 관련성(Relevance): 질문 의도에 맞는 답인지
* 완결성(Completeness): 빠진 내용 없이 충분한지
* 일관성/논리성(Coherence): 앞뒤가 맞고 논리적인지
* 안전성(Safety): 유해·편향·민감 정보 문제가 없는지
* 형식 준수(Instruction following): 요청한 형식·길이·톤을 지켰는지

### 평가 방법 ###

#### 1. 정답 기반 자동 지표 (참조 답안이 있을 때) ####

* BLEU, ROUGE, METEOR: 번역·요약에서 정답과의 n-gram 겹침 측정. 단순하지만 의미를 잘 못 잡습니다.
* BERTScore: 임베딩 기반으로 의미 유사도를 측정해 위 지표의 한계를 보완.
* Exact Match / F1: QA처럼 정답이 명확할 때.

#### 2. LLM-as-a-Judge (요즘 가장 널리 쓰임) ####

* 강력한 LLM(예: GPT-4급)에게 답변을 채점하게 하는 방식입니다.
* 방식: 점수 매기기(1~5점), 두 답변 비교(A vs B, pairwise), 기준별 루브릭 채점.
* 장점: 사람 평가에 근접하면서 저렴하고 빠름. 단점: 판정 모델의 편향(길이 편향, 자기 선호 편향, 위치 편향)에 주의해야 함.
* 대표 도구/벤치마크: MT-Bench, AlpacaEval, G-Eval.

#### 3. 사람 평가 (Human Evaluation) ####

여전히 골드 스탠다드입니다. 평가자가 직접 점수를 매기거나 두 답변 중 나은 것을 고릅니다(선호도 평가).
* RLHF의 기반이 되는 방식이기도 합니다.
* 단점: 비싸고 느리며, 평가자 간 일관성(inter-annotator agreement) 관리가 필요.

#### 4. RAG 특화 평가 (검색 기반 시스템일 때) ####

* Faithfulness(답변이 검색된 근거에 충실한지), Context Relevance(검색된 문서가 관련 있는지), Answer Relevance 등을 봅니다.
* RAGAS, TruLens 같은 프레임워크가 이걸 자동화해줍니다.

#### 5. 벤치마크 기반 평가 ####

MMLU(지식), GSM8K(수학 추론), HumanEval(코드), TruthfulQA(사실성) 등 표준 데이터셋으로 능력별 점수를 냅니다.

### 팁 ###
* 단일 지표에 의존하지 마세요. 자동 지표로 대량 스크리닝 → LLM-as-Judge로 세부 채점 → 중요한 케이스만 사람 평가, 이런 파이프라인이 흔합니다.
* 평가 기준(루브릭)을 먼저 명확히 정의하는 게 가장 중요합니다. 기준이 모호하면 사람이든 LLM이든 일관성이 떨어집니다.
* 실제 사용자 데이터로 만든 평가셋(golden set)을 유지하고, 모델·프롬프트 변경 시마다 회귀 테스트하듯 돌리면 품질 관리가 훨씬 수월해집니다.
