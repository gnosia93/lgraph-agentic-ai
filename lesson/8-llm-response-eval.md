LLM 답변 품질 평가는 크게 자동 평가와 사람 평가로 나뉘고, 실무에서는 둘을 섞어서 씁니다. 대표적인 방식들을 정리해드릴게요.

기준(무엇을 평가하나) 먼저 "품질"을 여러 축으로 나눕니다.

* 정확성(Accuracy / Factuality): 사실이 맞는지, 환각(hallucination)이 없는지
* 관련성(Relevance): 질문 의도에 맞는 답인지
* 완결성(Completeness): 빠진 내용 없이 충분한지
* 일관성/논리성(Coherence): 앞뒤가 맞고 논리적인지
* 안전성(Safety): 유해·편향·민감 정보 문제가 없는지
* 형식 준수(Instruction following): 요청한 형식·길이·톤을 지켰는지

### 평가 방법 ###

#### 1. 정답 기반 자동 지표 (참조 답안이 있을 때) ####

* BLEU, ROUGE, METEOR: 번역·요약에서 정답과의 n-gram 겹침 측정. 단순하지만 의미를 잘 못 잡습니다.
* BERTScore: 임베딩 기반으로 의미 유사도를 측정해 위 지표의 한계를 보완.
* Exact Match / F1: QA처럼 정답이 명확할 때.

#### 2. LLM-as-a-Judge (요즘 가장 널리 쓰임) ####

* 강력한 LLM(예: GPT-4급)에게 답변을 채점하게 하는 방식입니다.
* 방식: 점수 매기기(1~5점), 두 답변 비교(A vs B, pairwise), 기준별 루브릭 채점.
* 장점: 사람 평가에 근접하면서 저렴하고 빠름. 단점: 판정 모델의 편향(길이 편향, 자기 선호 편향, 위치 편향)에 주의해야 함.
* 대표 도구/벤치마크: MT-Bench, AlpacaEval, G-Eval.

#### 3. 사람 평가 (Human Evaluation) ####

여전히 골드 스탠다드입니다. 평가자가 직접 점수를 매기거나 두 답변 중 나은 것을 고릅니다(선호도 평가).
* RLHF의 기반이 되는 방식이기도 합니다.
* 단점: 비싸고 느리며, 평가자 간 일관성(inter-annotator agreement) 관리가 필요.

#### 4. RAG 특화 평가 (검색 기반 시스템일 때) ####

* Faithfulness(답변이 검색된 근거에 충실한지), Context Relevance(검색된 문서가 관련 있는지), Answer Relevance 등을 봅니다.
* RAGAS, TruLens 같은 프레임워크가 이걸 자동화해줍니다.

#### 5. 벤치마크 기반 평가 ####

MMLU(지식), GSM8K(수학 추론), HumanEval(코드), TruthfulQA(사실성) 등 표준 데이터셋으로 능력별 점수를 냅니다.

### 팁 ###
* 단일 지표에 의존하지 마세요. 자동 지표로 대량 스크리닝 → LLM-as-Judge로 세부 채점 → 중요한 케이스만 사람 평가, 이런 파이프라인이 흔합니다.
* 평가 기준(루브릭)을 먼저 명확히 정의하는 게 가장 중요합니다. 기준이 모호하면 사람이든 LLM이든 일관성이 떨어집니다.
* 실제 사용자 데이터로 만든 평가셋(golden set)을 유지하고, 모델·프롬프트 변경 시마다 회귀 테스트하듯 돌리면 품질 관리가 훨씬 수월해집니다.

## 평가 샘플 ##
각 케이스별로 평가 설계가 꽤 다릅니다. 상황별로 핵심 지표와 파이프라인을 정리해드릴게요.

### 1. RAG 챗봇 ### 
RAG는 검색(retrieval)과 생성(generation) 두 단계를 나눠서 평가하는 게 핵심입니다. 답이 틀렸을 때 검색이 문제인지 생성이 문제인지 구분해야 개선이 가능하거든요.

#### 검색 단계 평가 ####

* Context Recall: 정답에 필요한 문서를 실제로 가져왔는지
* Context Precision: 가져온 문서 중 관련 있는 것의 비율 (노이즈가 적은지)
* 전통 IR 지표: Hit Rate, MRR, NDCG (정답 문서 순위가 얼마나 위에 있는지)

#### 생성 단계 평가 ####

* Faithfulness / Groundedness: 답변이 검색된 근거에만 기반하는지 (환각 여부). RAG에서 가장 중요합니다.
* Answer Relevance: 질문 의도에 맞는 답인지
* Context Utilization: 가져온 근거를 실제로 잘 활용했는지
* 도구: RAGAS, TruLens, DeepEval이 위 지표를 자동화해줍니다. LLM-as-Judge로 Faithfulness를 채점하는 방식이 표준입니다.   

실무 팁: (질문, 정답, 근거 문서) 세트로 골든셋을 만들어두고, 임베딩 모델·청킹 전략·프롬프트를 바꿀 때마다 회귀 테스트하세요.

### 2. 요약 서비스 ###
요약은 "원문에 충실하면서 핵심을 잘 뽑았는지"가 관건입니다.

#### 핵심 축 ####

* Faithfulness(사실 일관성): 원문에 없는 내용을 지어내지 않았는지. 요약 환각이 실무에서 가장 큰 리스크입니다.
* Coverage(포괄성): 중요한 정보를 빠뜨리지 않았는지
* Conciseness(간결성): 불필요한 반복·군더더기가 없는지
* Coherence(가독성): 문장이 자연스럽고 논리적인지

#### 지표 ####

참조 요약이 있으면: ROUGE(관행적으로 많이 씀), BERTScore(의미 기반)
참조가 없을 때: LLM-as-Judge 루브릭 채점, QAG(요약에서 질문을 만들어 원문으로 답이 되는지 검증), SummaC/FactCC 같은 사실성 전용 모델
실무에서는 ROUGE는 참고용으로만 보고, 사실성 + LLM 루브릭 평가를 주로 씁니다.

### 3. 코드 생성 ###
코드는 "실행해서 맞는지" 검증할 수 있다는 게 다른 태스크와의 결정적 차이입니다.

#### 핵심 축 ####

* Functional Correctness: 실제로 동작하고 요구사항을 만족하는지 → 가장 중요
* 컴파일/실행 여부
* 코드 품질: 가독성, 스타일, 보안 취약점, 효율성

#### 지표 ####

* pass@k: 모델이 k개 답을 냈을 때 최소 하나가 테스트를 통과할 확률. 코드 평가의 사실상 표준 지표입니다.
* 테스트 스위트 기반 실행 평가: 준비된 유닛 테스트를 돌려 통과율 측정 (HumanEval, MBPP 벤치마크가 이 방식)
* 정적 분석: 린터, 타입 체커, 보안 스캐너(예: Semgrep)로 품질·취약점 점검
* LLM-as-Judge: 테스트로 못 잡는 가독성·설계 품질을 보조 평가
* 실무 팁: 가능한 한 자동 테스트로 검증되는 문제로 평가셋을 구성하세요. 실행 기반 평가가 LLM 채점보다 훨씬 신뢰도가 높습니다. 샌드박스 환경에서 격리 실행하는 것도 잊지 마세요.

### 공통 파이프라인 패턴 ###
세 케이스 모두 이런 흐름이 효율적입니다.

* 골든 평가셋 구축 (실제 사용자 데이터 기반)
* 저비용 자동 지표로 대량 스크리닝
* LLM-as-Judge로 세부 루브릭 채점
* 애매하거나 중요한 케이스만 사람 평가
* 모델·프롬프트 변경 시마다 회귀 테스트로 돌리기
