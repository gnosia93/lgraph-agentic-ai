## promptfoo 로 모델 평가하기 ##

LLM 애플리케이션을 개발하다 보면 이런 상황을 자주 마주하게 된다. 프롬프트를 수정했는데 기존보다 나아진 건지 확신이 없고, 모델을 바꿔보고 싶은데 하나하나 수동으로 비교하기엔 시간이 너무 걸린다. GPT-4o와 Claude를 놓고 어떤 게 우리 서비스에 더 맞는지 판단하려면, 같은 질문을 양쪽에 던지고 결과를 눈으로 비교하는 노가다를 반복해야 한다.
promptfoo는 이 과정을 자동화하는 오픈소스 LLM 평가 프레임워크다. 한마디로 표현하면, LLM을 위한 단위 테스트 프레임워크라고 할 수 있다.
소프트웨어 개발에서 코드를 변경할 때마다 단위 테스트를 돌려서 기존 기능이 제대로 동작하는지 확인하듯이, promptfoo 는 프롬프트나 모델을 변경시 평가를 돌려서 품질이 유지되는지 확인할 수 있게 해준다.


### 평가 기준 ###
promptfoo가 강력한 이유는 다양한 평가 방식을 지원한다는 점이다.
결정적(Deterministic) 평가는 코드로 판정하는 방식이다. 특정 문자열 포함 여부(contains), JSON 형식 검증(is-json), 응답 시간 제한(latency), 비용 제한(cost) 등이 있다. 정확한 기준이 있을 때 유용하다.  
모델 기반(Model-assisted) 평가는 LLM이 다른 LLM의 출력을 채점하는 방식이다. llm-rubric이 대표적인데, 자연어로 채점 기준을 작성하면 채점용 LLM이 해당 기준에 따라 pass/fail을 판정한다. "답변이 친절한가?", "기술적으로 정확한가?" 같은 주관적 기준도 평가할 수 있어서, 사람이 일일이 채점하는 수고를 줄여준다.

### 어떤 상황에서 유용한가? ###
* 프롬프트 엔지니어링: 프롬프트를 수정할 때마다 기존 품질이 유지되는지 자동 검증
* 모델 비교: 동일한 테스트셋으로 여러 모델의 품질, 비용, 속도를 객관적으로 비교
* CI/CD 통합: 프롬프트 변경을 코드 변경처럼 관리하고, PR(Pull Request) 마다 자동 평가 실행
* 레드팀 테스트: 67개 이상의 보안 공격 플러그인으로 LLM 취약점 스캔

Promptfoo의 가치는 LLM judge 자체보다는, 여러 모델 × 여러 테스트 케이스 × 여러 평가 기준을 한 번에 자동으로 돌리고 비교할 수 있는 프레임워크를 제공한다는 점이다.
이번 섹션에서는 Promptfoo 을 이용하여 AWS Bedrock 에서 제공되는 모델의 성능을 측정한다.

### 6. 테스트 시나리오 작성 ###
```
cat <<EOF > promptfooconfig.yaml
description: AWS Bedrock 모델 비교 평가

prompts:
  - |
    다음 질문에 한국어로 답변해줘.
    정확하고 간결하게 답변할 것.

    질문: {{question}}

providers:
  # Claude Sonnet 4
  - id: bedrock:apac.anthropic.claude-sonnet-4-20250514-v1:0
    config:
      region: ap-northeast-2
      temperature: 0
      max_tokens: 1024

  - id: bedrock:apac.anthropic.claude-3-haiku-20240307-v1:0
    config:
      region: ap-northeast-2
      temperature: 0
      max_tokens: 1024

  # Amazon Nova Pro
  - id: bedrock:apac.amazon.nova-pro-v1:0
    config:
      region: ap-northeast-2
      interfaceConfig:
        temperature: 0
        max_new_tokens: 1024

# 모든 test에 적용되는 공통 테스트 기준을 명시.
defaultTest:   
  options:
    provider:   # 평가 LLM(judge LLM) 설정
      id: bedrock:apac.anthropic.claude-3-haiku-20240307-v1:0
      config:
        region: ap-northeast-2
        temperature: 0
  assert:
    # cost 는 지원하지 않으므로 주석처리한다. 
    #- type: cost
    #  threshold: 0.05
    - type: latency
      threshold: 10000
    # 한국어로 답변했는가
    - type: llm-rubric
      value: "답변이 한국어로 작성되어 있는가?"
      metric: 언어
    # 기술적 정확성
    - type: llm-rubric
      value: "기술적으로 틀린 내용이 없는가?"
      metric: 정확성
    # 구조화된 답변인가
    - type: llm-rubric
      value: "답변이 논리적으로 구조화되어 있고, 핵심을 먼저 말하는가?"
      metric: 구조
    # 실용성
    - type: llm-rubric
      value: "실무에서 바로 활용할 수 있는 구체적인 예시나 코드가 포함되어 있는가?"
      metric: 실용성
    # 간결성 (너무 길지 않은가)
    - type: javascript
      value: "output.length < 3000 ? 1.0 : Math.max(0, 1 - (output.length - 3000) / 5000)"
      metric: 간결성

tests:
  - vars:
      question: "쿠버네티스에서 파드와 디플로이먼트의 차이점은?"
    threshold: 0.7
    assert:         # 아래 기준 1개 + defaultTest 6개에 대해서 대해서 test 를 수행한다.      
      - type: llm-rubric
        value: "파드와 디플로이먼트의 차이를 정확히 설명하고, 실무 관점의 예시가 포함되어 있는가?"
    
  - vars:
      question: "REST API와 GraphQL의 장단점을 비교해줘"
    threshold: 0.7
    assert:
      - type: llm-rubric
        value: "두 기술의 장단점이 균형 있게 설명되어 있는가?"
      - type: contains
        value: "GraphQL"

  - vars:
      question: "Docker 컨테이너와 가상머신의 차이는?"
    threshold: 0.7
    assert:
      - type: llm-rubric
        value: "기술적으로 정확하고, 초보자도 이해할 수 있는 설명인가?"
EOF
```
promptfoo 의 테스트 시나리오는 다음과 같이 총 4개의 섹션으로 이뤄져 있다.

#### 6-1. 기본 정보 및 프롬프트 정의 ####
* description: 이 테스트 프로젝트의 목적을 나타내는 이름입니다. 웹 대시보드나 결과 리포트에 표시됩니다.
* prompts: 테스트 대상 모델들에게 입력할 공통 프롬프트(지시문)입니다. {{question}} 부분에 하단의 테스트 케이스들이 변수로 주입됩니다.

#### 6-2. 평가 대상 모델 (Providers) 설정 ####
* providers: 답변 품질을 비교할 AWS Bedrock의 모델 목록입니다. (Claude Sonnet 4, Claude 3 Haiku, Amazon Nova Pro 총 3종)
* config: 각 모델의 호출 옵션입니다. 모든 모델의 temperature를 0으로 설정하여 답변의 일관성(무작위성 배제)을 확보하고, 아시아 태평양(서울) 리전(ap-northeast-2)을 바라보도록 설정했습니다. Amazon Nova 모델의 경우 토큰 및 온도를 설정하는 하위 속성명이 interfaceConfig로 정의되어 있습니다.
* max_tokens - 모델이 한 번에 생성할 수 있는 최대 출력 토큰 수

#### 6-3. 공통 검증 기준 (Default Test) ####
이 섹션은 모든 테스트에서 공통적으로 적용되는 테스트 케이스를 작성하는 부분이다. 테스트 케이스를 나열한 tests 섹션의 "쿠버네티스에서 파드와 디플로이먼트의 차이점은?"
의 경우 설정상 하나의 assert 만 있지만 defaultTest 에 설정된 6개의 항목과 더불어서 총 7개의 테스트가 실행되게 된다.

#### 6-4. 개별 테스트 케이스 (Tests) ####
질문 데이터와 해당 질문에만 적용되는 특화 검증 기준입니다. 모든 테스트는 threshold: 0.7 조건이 있어, 공통 기준과 개별 기준을 합산한 점수가 70% 이상이어야 합격(Pass) 처리됩니다.
이 예제의 경우  쿠버네티스 질문, REST API vs GraphQL 질문 그리고  Docker vs VM 질문으로 구성되어져 있으며 각기 다른 평가 기준을 가지고 있다.
threshold 0.7 의 경우 다음과 같은 의미를 가지게 된다. << 추가 한다. >>

### 8. 테스트 케이스(Assert 타입 정리) ###

#### 결정적(Deterministic) 평가 — LLM 없이 코드로 판정 ####

| 카테고리 | assert 타입 | 설명 |
|---------|------------|------|
| 문자열 | equals | 정확히 일치 |
| | contains | 특정 문자열 포함 |
| | icontains | 포함 (대소문자 무시) |
| | contains-any | 목록 중 하나라도 포함 |
| | contains-all | 목록 전부 포함 |
| | starts-with | 특정 문자열로 시작 |
| | regex | 정규식 매칭 |
| 형식 검증 | is-json | 유효한 JSON (스키마 검증 가능) |
| | contains-json | JSON 포함 여부 |
| | is-html | 유효한 HTML |
| | is-sql | 유효한 SQL |
| | is-xml | 유효한 XML |
| 성능 | latency | 응답 시간 임계값 (ms) |
| | cost | 비용 임계값 ($) |
| | perplexity | 퍼플렉시티 임계값 |
| 텍스트 유사도 | rouge-n | ROUGE-N 점수 |
| | bleu | BLEU 점수 |
| | levenshtein | 편집 거리 |
| | meteor | METEOR 점수 |
| 커스텀 | javascript | JS 함수로 판정 |
| | python | Python 함수로 판정 |
| | webhook | 외부 웹훅으로 판정 |
| 기타 | is-refusal | 모델이 거부 응답을 했는가 |
| 에이전트 | trajectory:tool-used | 특정 도구를 호출했는가 |
| | trajectory:tool-sequence | 도구 호출 순서가 맞는가 |
| | trajectory:tool-args-match | 도구 호출 인자가 맞는가 |
| | trajectory:step-count | 스텝 수 검증 |

#### 모델 기반(Model-assisted) 평가 — LLM이 판정 ####

| 카테고리 | assert 타입 | 설명 |
|---------|------------|------|
| 범용 | llm-rubric | 자유 기준으로 LLM이 채점 |
| | g-eval | Chain-of-thought 기반 평가 |
| | similar | 임베딩 코사인 유사도 |
| | classifier | 분류기로 판정 |
| | select-best | 여러 출력 중 최고 선택 |
| | factuality | 사실 관계 검증 |
| | model-graded-closedqa | OpenAI Closed QA 방식 평가 |
| RAG 특화 | answer-relevance | 답변이 질문과 관련 있는가 |
| | context-faithfulness | 답변이 컨텍스트에 충실한가 |
| | context-recall | 정답이 컨텍스트에 포함되어 있는가 |
| | context-relevance | 컨텍스트가 질문과 관련 있는가 |
| 에이전트 | trajectory:goal-success | 에이전트가 목표를 달성했는가 |
| 대화 | conversation-relevance | 대화 흐름에서 응답이 관련 있는가 |

> 모든 타입 앞에 `not-`을 붙이면 반대로 동작한다. 예: `not-contains`, `not-regex`
